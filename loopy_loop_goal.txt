# Goal

Implement the complete Ultimate Memory library by executing the build roadmap in
plan/plans/roadmap.md — phases 0 through 8, in order, to done.

## How to work (double loop)

- This goal drives the PM parent session (planner + dispatcher). The parent
  manages the roadmap and never implements: the planner selects one work package
  at a time, the dispatcher starts a child session for it, and the planner
  reviews the child's evidence before accepting.
- plan/plans/roadmap.md is the sequencing authority. The phase files
  (plan/plans/phase-0-*.md … phase-8-*.md) define each phase's entry gates, work
  packages (WP-N.x) with Reads columns and acceptance criteria, and exit
  criteria. The phase files are the backlog; PM state tracks selection and
  review, never a parallel copy.
- Each child session implements exactly one WP (or a small WP group the phase
  file itself couples) on the inner_outer_eval workflow set. The child's goal
  must be self-contained: WP id, Reads column, acceptance criteria, delivery
  and eval expectations (the dispatcher writes it that way).
- A work package is executed after reading the documents named in its Reads
  column (plus the repo conventions in the system rules). The design documents
  are binding; where a plan and a design appear to disagree, the design wins
  and the discrepancy is recorded.
- WP status columns (planned | in-progress | done | blocked(<gate>)) are
  flipped by the child, committed together with the work; the planner verifies
  the flip during review.
- Respect the roadmap's gate register: a phase must not start while one of its
  entry gates is open. If a gate is blocked on a human decision, record it and
  continue with unblocked work.
- All implementation lands via branch → PR → checks → merge. CI (ruff, pyright,
  pytest, coverage) must be green before merge.

## Eval focus

- Keep eval-banana checks central to the loop. The curated repo check set in
  plan/implementation_evals/eval_checks/ (design-conformance checks, maintained
  in-repo) is the base: child eval workflows run it, don't invent overlapping
  checks, and add session-scoped checks that verify the WP's specific behavior —
  phase exit criteria, the designs' own contract tests (the retrieval S-battery
  scenarios, CI invariants, planted canaries). Checks must verify behavior,
  never merely that files exist.
- A child reports goal_met only when both the curated set and its session set
  pass.

## Completion criteria

The goal is met only when all of the following are true:

- Every phase file (phase-0 through phase-8) shows all of its work packages done
  and its exit criteria met, with evidence linked from the PM session's
  project_state/finished.md.
- Phase 8's benchmark exit criteria pass and the results are recorded.
- CI is green on main and all implementation work was merged through PRs (no
  orphaned implementation branches).
- The PM session's project_state/finished.md contains enough evidence per phase
  for a reviewer to verify completion without re-deriving it.

## Notes

- Phase 0 is already partially complete on main (package scaffold, tooling
  config, CI with coverage, the pydantic-settings config convention). Read the
  phase-0 file's statuses and continue from there — do not redo finished work.
- Numbers in designs marked as "starting points to measure" are exactly that:
  measure on the golden set / corpus slices before hard-coding; record
  measurements where the design's spike list asks for them.
- A child session that stops with workflow_failure_cap or unresolvable_error is
  evidence, not the end of the run: review its recorded blocker, then rescope,
  re-dispatch with a better goal, or record the exact blocker and move to
  unblocked work.
