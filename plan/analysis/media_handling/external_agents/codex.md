# Media Handling — Independent Design Analysis

## Position

The current design has the right top-level policy and an incomplete execution model.

The right policy is already explicit in `requirements_v3.md` §Retrieval, D51, and
`e0_files_design.md` §2/§5: agents should normally encounter a Markdown rendering, while the
immutable original remains deliberately reachable through a raw pointer. A 2 GB video must not be
copied into `artifacts/media/`, and a transcript must not be mistaken for a lossless substitute for
the recording. The raw mount is therefore not a fallback storage detail; it is one half of the
evidence contract.

What is missing is a media-capable form of the other half. E0 has no standalone image, audio, or
video routes. Its converter contract has only a page-oriented source map. E1 provenance ends at
`{page?, bbox?}`. D32 grounds a claim in text generated by the converter but cannot say where that
text came from in the recording or image. D54 defines an extraction basis from raw content and the
E2 extractor version, omitting the ASR/VLM generation that produced the text E2 actually read. P1
can search the generated text, but cannot retrieve an inadequately described visual or acoustic
event directly.

My recommendation is:

> Treat every media input as one document lineage whose immutable raw asset is the sensory source,
> whose current **representation generation** is the Markdown-first, model-derived reading of that
> source, and whose claims carry both a text anchor in that reading and a typed source locator into
> the immutable raw asset.

The raw asset and the derived representation have different jobs. Raw bytes are the audit target
and the material a multimodal consumer may inspect. `document.md` is the common text surface that
E1 can blockize, E2 can extract from, P1 can search, and P3 can summarize. Neither replaces the
other. `media/` contains only derived supporting assets such as keyframes, crops, thumbnails, and
optional timed-transcript interchange files; it never contains a second copy of a whole-file
original.

This requires design changes, not a new plane or a parallel media pipeline. Media is an E0 input
modality and a P1 retrieval target. The existing E0→E1→E2→E3 architecture should remain one
architecture.

## 1. Problem anatomy

### 1.1 What the design already gets right

Several existing decisions compose well:

- `e0_files_design.md` §2 separates immutable raw objects from mountable artifacts and restricts
  `media/` to derived material. This prevents expensive duplication of whole recordings while
  retaining extracted figures and keyframes next to the Markdown that refers to them.
- D51 and `retrieval_design.md` §7 make raw originals read-only, off the navigation path, and
  reachable only through explicit pointers. This implements “Markdown-first” as a navigation rule,
  not as denial of raw access. S56 checks the embedded-figure path; S59 checks the whole-recording
  path.
- `e1_chunks_design.md` D57 establishes `document.md` as one immutable character coordinate system.
  This is exactly the right substrate for a transcript or image description, provided a second map
  connects those characters to source time or source regions.
- PageIndex always yields a structure, even if only a synthetic root. Standalone images and short
  recordings therefore do not need a special exception to enter E1.
- D54’s distinct-lineage counting rule prevents multiple captions, transcript turns, keyframes, or
  repeated mentions from pretending to be independent corroborating sources.

The owner's requirement is therefore mostly a clarification of these decisions:

```
find and reason cheaply                       inspect or challenge the derivation
            |                                               |
            v                                               v
P3 stub -> document.md -> blocks/chunks/claims     explicit pointer -> immutable raw asset
              |                                               ^
              +-- typed source map: time/page/region ----------+
```

### 1.2 The gaps are coupled

The missing router rows are not isolated adapter work. Adding “MP3 → transcript” without changing
provenance and lifecycle would create a misleading system:

1. E2 could extract a claim from a transcript paragraph.
2. D32 could prove that the paragraph exists in `document.md`.
3. Nothing could prove which seconds of the MP3 the paragraph represents.
4. An ASR upgrade could change the paragraph while the raw `content_hash` and E2
   `extractor_version` stayed constant.
5. Old and new claims would have no clean representation-basis transition under D54.

Likewise, adding a VLM caption without an epistemic label would let “a model described pixels this
way” masquerade as “the source author asserted this.” Raw reachability makes the error auditable,
but it does not prevent downstream consumers from misunderstanding it.

The solution must bind conversion routes, artifact shape, source locators, grounding, testimony
currency, and retrieval together.

## 2. Recommended conceptual model

### 2.1 Three objects, not one overloaded “media file”

Use three concepts with explicit identities:

1. **Source asset** — the immutable bytes at
   `raw/<doc_id>/<content_hash>/original.<ext>`. This is the source-of-record for sensory content.
   It belongs to a document version and is addressed by an immutable raw handle. Duration,
   dimensions, codec, track layout, and normalized embedded metadata are technical facts about this
   asset, recorded by deterministic inspection where possible.
2. **Representation generation** — one persisted conversion of that asset under a pinned route and
   pinned component versions. It consists of `document.md`, a generalized source map, a conversion
   manifest, PageIndex/block sidecars, and a derived-media manifest. It is what E1 and E2 consume.
   ASR output, OCR text, VLM descriptions, diarization, and video shot notes belong to this
   generation.
3. **Derived media asset** — a regenerable child such as an extracted figure, image crop, thumbnail,
   keyframe, contact sheet, waveform, or optional `.vtt` transcript. It lives in `media/`, has its
   own hash and derivation provenance, and points to the source region/time from which it came. It
   is not an independent document lineage unless it was separately ingested from a source with its
   own connector-native identity.

This distinction resolves an ambiguity in `e0_files_design.md` §2. That section lists transcripts
inside `media/`, while E0 §3 says the converter's Markdown is the downstream body. A transcript may
also be emitted as `.vtt` or JSON for interoperability, but the **canonical ingestible transcript
text must appear in `document.md`**. Text that exists only in `media/transcript.vtt` is invisible to
the blockizer, E2, ordinary P1 text search, and D32's character anchors.

### 2.2 `document.md` is the testimony carrier, with a necessary qualification

Operationally, all media-derived text should enter the same Markdown testimony surface. Semantically,
not all such text has the same relationship to the source:

- **Literal/source expression**: embedded human captions, OCR-visible text, or a transcript of
  speech. These are fallible renderings of symbols or utterances present in the source.
- **Perceptual description**: a VLM says that an image shows a red valve, or that a video shot shows
  Alice entering a room. This is a model observation about pixels, not an assertion made by the
  image or its author.
- **Interpretive description**: “the speaker sounds hesitant,” “the crowd is hostile,” or “the chart
  indicates strong growth.” These are especially model-mediated and must never silently become
  literal source testimony.

The current claim vocabulary can remain; this does not require fact/opinion/prediction claim types,
which the requirements reject. It does require queryable **grounding/mediation metadata** on a claim
occurrence or its evidence link:

```
source_modality: text | image | audio | video
derivation_kind: passthrough | ocr | embedded_caption | asr | acoustic_description |
                 vlm_description | multimodal_fusion
evidence_mode: source_expression | model_observation | model_interpretation
derivation_run_id: <immutable representation/component run>
source_locators: [<typed locators into the raw version>]
```

This metadata explains how the claim was obtained; it is not a verdict on whether the claim is
true. `evidence_count` should continue to count distinct source lineages, not derivation runs. A
caption and an ASR transcript from the same video are two views of one source, not two witnesses.

### 2.3 Core invariants

The media design should bind these invariants:

1. Every accepted input, including a standalone photo or recording, produces `document.md` and a
   document structure.
2. Every textual passage eligible for E2 appears in `document.md`, not only in a sidecar.
3. Every media-derived passage maps to at least one immutable raw locator, at an honestly reported
   precision.
4. Every raw locator names a document **version**, never merely a living lineage or a P3 view path.
   Historical claims must not jump into the current replacement file.
5. Every representation-generating model/configuration is pinned and recorded. A re-run replays
   stored output; it never silently calls a nominally identical nondeterministic model again.
6. A representation upgrade of unchanged raw bytes is a processing change, not a source edit. D54's
   processing-driven support-withdrawal semantics apply.
7. The raw asset is available through both the raw mount and API/CLI by the same logical handle.
8. Derived media is regenerable and deletable with its parent version. Raw remains the audit root.

## 3. Converter routes and artifact contract

### 3.1 Generalize the converter contract

`e0_files_design.md` §3 and D38 should replace the page-specific contract with a source-map
contract:

```
convert(bytes, mime, hints) -> {
  markdown,
  source_map[],          # document.md char ranges -> typed raw locators
  derived_assets[],      # manifest entries for files placed under media/
  conversion_manifest   # route graph, versions, coverage, warnings, output hashes
}
```

`page_map` becomes one supported `source_map` locator type rather than the universal name. The
contract remains capability-tiered: a PDF route may provide page and bounding box; audio must
provide time ranges; a video passage may provide time plus a frame region; a plain email may have
only a source-body range. Missing precision is represented explicitly, never fabricated by
interpolating offsets.

`media[]` should become a manifest of derived assets, not merely “extracted images: bytes, page,
caption.” Each entry needs an ID, media kind, MIME type, content hash, artifact-relative path,
parent raw locator(s), producer run, and the Markdown spans/links that refer to it.

### 3.2 Bind the missing MIME routes

The E0 router table should include at least these composable routes, selected by MIME sniffing rather
than filename extension:

| Input | Required derivation | `document.md` content | Derived `media/` content |
|---|---|---|---|
| Standalone image | deterministic metadata inspection; OCR; region-aware VLM description | source metadata; overview; visible text separated from description; region/detail notes; limitations; raw pointer | thumbnail and useful crops/overlays; never a duplicate full-resolution original |
| Audio with speech | ASR with segment timestamps; word timestamps where supported; language detection; diarization for multi-speaker material | timed speaker turns, original-language transcript, acoustic events/notes in separately labeled sections | optional `.vtt`/timed JSON, waveform, small previews if useful |
| Non-speech or mixed audio | acoustic event segmentation/captioning in addition to any ASR | timed descriptions such as music, alarms, silence, applause, with interpretation labels | optional waveform/event visualization |
| Video | track inspection; ASR + diarization; shot-boundary detection; adaptive keyframes; keyframe OCR; VLM shot/scene notes; shared timeline | timed transcript interleaved or cross-linked with scene/shot headings, visible text, and visual notes | keyframes, thumbnails/contact sheets, useful crops; optional timed transcript interchange |
| Animated image | treat as short video, not as one still | timed frame/scene descriptions and visible text | sampled frames/keyframes |

Existing embedded captions/subtitles should be preserved and labeled as source-provided. They should
not be silently overwritten by ASR. A route may also generate ASR for comparison, but the two
representations need distinct provenance. Original-language text must remain available when an
optional translation is emitted; translation is another model-mediated derivation, not a
replacement.

For diarization, bind the useful capability without overclaiming identity. The route should assign
stable source-local labels such as `Speaker 1` and `Speaker 2`, with segment-level confidence and
overlap markers. Mapping `Speaker 2` to “Bob” is a separate attribution decision. A participant list
does not prove which voice is Bob. Unless metadata, an explicit self-identification, or a configured
voice-identity verifier supports the mapping, E2 must keep the local speaker label. This means some
attributed stances will correctly fail D59's resolved-holder condition rather than being assigned to
the wrong person.

### 3.3 Shape of the Markdown

The rendering should be ordinary readable Markdown, not a private serialization. Examples of useful
shapes are:

```
# Steering call — 2026-07-10

Source: audio · duration 00:48:12 · language en
Raw: <immutable raw handle>
Representation: ASR run asr-... · diarization run diar-...

## Transcript

**[00:14:32–00:14:45] Speaker 2:** We will move the cutover to 18 July.

## Acoustic events

**[00:14:44–00:14:47]** Overlapping speech; the final words are low confidence.
```

and for an image:

```
# IMG_1842

Raw: <immutable raw handle>

## Visual description

[Model observation] A whiteboard shows a deployment timeline with three columns.

## Visible text

[OCR, region r3] "Cutover — 18 Jul"

## Regions

![Region r3](media/region-r3.png)
```

The bracketed time and region references are display forms backed by machine-readable entries in the
source map. They must not be the only provenance. Frontmatter should carry the logical raw handle,
the raw-mount-relative path, representation ID, modality, duration/dimensions, derivation kinds, and
quality/coverage warnings. It should not contain environment-specific absolute mount paths.

### 3.4 Landing rule

Use a simple rule:

- If text should be chunked, searched as testimony, or offered to E2, it belongs in
  `document.md`.
- If it is a generated binary view that helps a multimodal reader inspect the source, it belongs in
  `media/` and is linked from `document.md`.
- If it is structured provenance or a machine interchange form, it belongs in `conversion.json` or
  a referenced sidecar and is summarized/linked from `document.md`.
- If it is the whole-file input, it belongs only in raw storage and is referenced explicitly.

This preserves the derived-only rule for `media/` while removing the present ambiguity around
transcripts.

## 4. Blocks, sections, and standalone media documents

### 4.1 The structure layer largely works

`e1_chunks_design.md` §2–§4 does not need a parallel media blockizer. Once media derivations are
rendered as clean GFM, the shared deterministic blockizer can produce headings, paragraphs, lists,
tables, and captions exactly as it does for office documents. PageIndex can draw semantic topic
sections over a transcript; the snap-to-block-grid rule still guarantees a well-formed partition;
semchunk can pack whole turns/paragraphs into passage-sized chunks.

This reuse is important. A separate “audio chunk” identity system would duplicate D56's diff,
packing, extraction, and search machinery and would make mixed-media documents difficult.

### 4.2 Required structural refinements

The current role enum in `e0_files_design.md` §4 lacks media roles. Add roles such as:

```
transcript | scene | shot_notes | visual_description | visible_text |
acoustic_events | media_metadata
```

These are section roles, not new claim types. They let E2 and P1 distinguish literal transcript/OCR
from VLM descriptions and low-value technical metadata. `body` remains a valid fallback.

Do not create a Postgres `document_sections` row for every ASR word or speaker turn. That would turn
the compact section index into a transcript-segment table. Timed segments belong in the source-map
sidecar; semantic sections should be chapters, scenes, agenda items, or topic spans. A converter may
render deterministic scene/time headings, and PageIndex may add semantic hierarchy above them.

The block record should generalize `page`/`bbox` to `source_anchors[]`. A transcript paragraph is
still a normal paragraph block, but its block record can point to a time interval. A video shot note
can point to a time interval plus one or more frame regions. One block may require multiple anchors,
especially when it fuses a spoken statement with a visible slide.

### 4.3 Standalone images need a document shape

Yes: every standalone image must become a document, not merely an entry in somebody else's
`media/` directory. Its `document.md` can be short, and PageIndex may reduce it to a synthetic root,
but it still needs:

- immutable lineage/version identity;
- a raw pointer;
- normalized metadata and capture time if present;
- separate OCR and VLM sections;
- spatial region locators;
- PageIndex root summary and placement hint;
- blocks/chunks for P1 and E2.

An image extracted from a PDF remains a derived child of that PDF and retains a page/bbox parent
locator. It becomes a separate lineage only when independently ingested or when the source connector
provides a stable independent identity. This prevents one PDF figure from counting as a second
source supporting the PDF's own claim.

## 5. Temporal and spatial provenance

### 5.1 Add a typed source-locator union

E1 §2 currently calls `{page?, bbox?}` “acceptable” tiered provenance. It is acceptable for current
routes but not for AV. Replace it with a union that can express:

```
SourceLocator =
  | {kind: page, page: int, bbox?: normalized_rect, precision: page | region}
  | {kind: image_region, region: normalized_rect, image_width, image_height}
  | {kind: time, start_ms: int, end_ms: int, track_id?, precision: word | segment | shot}
  | {kind: video_region, start_ms, end_ms, region: normalized_rect, keyframe_id?}
  | {kind: source_range, ...}  # pageless literal formats when available
```

Every locator also carries or inherits `version_id`, `content_hash`, and the raw handle. Use integer
time in a declared time base, not a formatted string or frame number alone; variable-frame-rate
video makes frame numbers non-portable. Human-facing `00:14:32` is a rendering of `start_ms`.

The source map connects a `document.md` character interval to one or more locators. ASR adapters
must provide at least timed segments. Word alignment yields a tighter claim locator when available;
without it, the system honestly returns the enclosing segment and `precision=segment`. It must never
pretend that proportional character interpolation is word timing.

### 5.2 Carry the locator through grounding

D32's dual-field grounding remains useful but becomes a two-hop chain:

```
claim_text
  -> source_span + char offsets in document.md        exact, deterministic
  -> source_map intersection                         exact mapping operation
  -> raw version + time/page/region locator           converter precision, disclosed
```

The deterministic D32 anchor proves that the claim derives from a real slice of the current
representation. It does **not** prove that the ASR heard correctly or that the pixels entail a VLM
description. D32's sampled independent audit must therefore be modality-aware: an auditor of an ASR
claim listens to the referenced interval; an auditor of a VLM claim inspects the referenced raw
frame/region; an auditor of OCR checks the pixels. Auditing only the derived Markdown would grade the
converter against its own output.

Claims whose grounding combines transcript and visual evidence need multiple locators, not one
arbitrary “primary” timestamp. The response envelope can show a concise primary locator while the
hydration handle exposes all anchors.

### 5.3 Make deep links real on every surface

P3 view paths are unsuitable provenance: they can reorganize, and a lineage's stable stub points to
its current version. A claim from a 2024 version must instead hold an immutable version/raw handle.

`document.md` should render each timed/region label as a stable logical source link and also expose
the raw-mount-relative path in frontmatter. The consumption skill maps that relative path under the
deployment's raw mount. API/CLI parity needs a locator-aware operation—either an extension of
`hydrate depth=bytes` or a `source_open` primitive—that accepts the same handle plus
`start_ms/end_ms` or region and returns a seekable source/codec-aware segment. A naive HTTP byte range
is not enough for arbitrary video codecs. Generating a bounded stream or ephemeral clip is a serving
operation, not a new source artifact or an LLM query.

The retrieval envelope in §5 should include source locators when evidence is hydrated, alongside the
existing provenance handle. S59 should be strengthened from “can fetch the original” to “can move
from a transcript claim to the exact raw interval, mounted and unmounted.” Add the equivalent image
scenario: a claim from a VLM/OCR passage opens the cited raw region, not merely a thumbnail.

### 5.4 Do not confuse three kinds of time

The media timeline is a source coordinate, not either temporal axis in D41:

- `start_ms=872000` means where in the file the evidence occurs.
- `claim_valid_from=2026-07-18` means when the source says the cutover is valid in the world.
- `ingested_at`/transaction time means when the system learned it.

Schemas, API field names, and the consumption skill must state this distinction. Calling both values
“timestamp” invites incorrect as-of queries.

## 6. Epistemics and lifecycle of derived testimony

### 6.1 Version stamps and raw access are necessary, not sufficient

`converter_version` plus an audit path answers “which tool produced this, and can I inspect the
source?” It does not answer:

- whether the passage is literal OCR/ASR or an interpretive VLM description;
- which ASR, diarizer, VLM, prompt, model revision, and sampling policy participated;
- which seconds/regions were examined or skipped;
- how confident the speech/speaker alignment was;
- whether a video description covered every shot or only periodic samples;
- whether the claim was ever independently checked against raw media.

`conversion.json` therefore needs a persisted derivation graph: adapter and model revisions,
prompt/config hashes, decoding parameters, language/track choices, component input/output hashes,
coverage statistics, quality warnings, and parent/child asset locators. One opaque
`converter_version` remains useful as the routing/invalidation knob, but it cannot be the entire
audit record.

Representation frontmatter and evidence hydration should expose concise mediation labels and
warnings. Segment confidences are diagnostics, not mathematical fact confidence; they should not be
silently multiplied into `evidence_count`.

### 6.2 The current extraction-basis definition misses converter changes

`evidence_lifecycle_design.md` §1 defines extraction basis as `(content_hash,
extractor_version)`. That is incomplete even for OCR and becomes untenable for ASR/VLM. The E2
extractor does not read raw bytes; it reads a converter-generated representation. An ASR model bump
can change every sentence while both coordinates in the present basis remain unchanged.

Introduce an immutable **document representation** beneath a document version:

```
document version (source bytes, source_modified_at)
  -> representation generation A (converter route/config/run, document.md, source map, media)
  -> representation generation B (new ASR/VLM generation, immutable alongside A)
```

`document_versions` should point to a current `representation_id`. Artifact paths must include that
ID or an equivalent representation hash, for example:

```
artifacts/<doc_id>/<content_hash>/<representation_id>/document.md
```

This fixes a latent immutability collision in E0 §2: today a converter bump is allowed to change
`document.md`, but the canonical path contains only `doc_id/content_hash`. Rewriting that path would
invalidate old claim offsets and contradict the claim that `document.md` is immutable and
content-hash-addressed.

The current testimony basis should become conceptually:

```
(source document version, representation generation, E2 extractor generation)
```

The representation identity should cover the Markdown plus all semantics that affect grounding and
extraction: source-map/segment labels, speaker attribution, derivation modes, and linked-asset
manifest. It should not rely only on nominal model names. Output hashes make replay and exact
identity testable.

### 6.3 Upgrade semantics

The transition rules then become clean:

- **Raw content changes**: a new document version; apply `snapshot` versus `living` semantics from
  D55.
- **ASR/OCR/VLM/converter changes on unchanged raw**: a new representation generation of the same
  source version. This is processing-driven re-transcription, exactly like an E2 extractor bump.
  Atomically switch current testimony only after conversion, E1, and E2 complete. Old claims become
  non-current by representation coordinates. If a fact loses all support, raise
  `support_withdrawn`; do not retract it as though the source acted.
- **Representation output is semantically identical**: reuse blocks, claims, and embeddings by
  exact stable hashes. Still attach the occurrence to the new representation/derivation run so an
  audit does not claim that an old model produced the current occurrence.
- **Only source locators improve**: text claims may be reused, but the current occurrence must point
  to the new source map. This is another reason mediation/source locators belong on the
  claim-occurrence/evidence mapping rather than solely on an immutable `claims` row.

E1 §7/D56's `extraction_input_hash` should include a stable representation-semantics hash in addition
to block text, neighbors, stable headers, extractor version, and structurer version. An opaque
converter version need not force E2 work if text and relevant semantics are byte-identical; changed
speaker labels, mediation kinds, or grounding maps must not be invisible merely because the
paragraph characters stayed the same.

The thin `chunk_claims` occurrence map in lifecycle §6 should consequently identify the
representation as well as the document version and chunk. This preserves exact `claims_as_of`
behavior across both source versions and converter generations.

### 6.4 Model-mediated evidence must remain visible downstream

Raw access is an audit capability, not an epistemic policy. Without mediation metadata, P1, E3, K,
and a consuming agent see only fluent text. The representation kind and derivation run should be
available in:

- claim/evidence hydration;
- the D49 evidence-grain response envelope;
- K compile inputs and citations when a page relies on perceptual descriptions;
- review cases raised by representation upgrades;
- the consumption skill's explanation of source expression versus model observation.

This does not require automatically distrusting every VLM claim or creating a second fact layer.
It makes the evidence legible. K3 admission and other confidence policies can then require stronger
support or raw audit for high-impact claims without corrupting the meaning of `evidence_count`.

## 7. Search: derivation text is the default, not the whole capability

### 7.1 What text search covers well

The current P1 architecture is a good default for speech and described visual content:

- ASR makes spoken words available to BM25/FTS and semantic chunk/claim search.
- OCR makes signs, slides, screenshots, and chart labels lexically retrievable.
- VLM scene and image descriptions make common visual concepts searchable by text.
- E2 claims remain the fine-grained semantic “needle” index described in E1 §5.

For “where did anyone say the cutover is 18 July?”, direct audio embeddings add little over a good
transcript. Text remains the cheap-first route.

### 7.2 Text alone has a hard recall boundary

Descriptions are selective. A VLM may omit the small red connector that a later query cares about.
ASR says nothing about a melody, alarm, engine noise, cadence, or nonverbal exchange. Video shot
notes cannot mention every object or visual relationship. Raw reachability does not solve discovery:
an agent cannot decide to open an unknown file it never retrieved.

The complete design should therefore add a P1 **media-segment semantic target**, not replace text
search:

```
search(channel=semantic, target=media_segments, query=<text, image, audio>, filters=..., k=...)
```

Rows are derived projection entries keyed to immutable locators:

- one embedding for a standalone image, plus region embeddings only where the converter created
  meaningful regions;
- visual embeddings for adaptive video keyframes/shots, not every frame;
- acoustic embeddings for bounded audio segments/events, separate from transcript embeddings;
- optional fused audio-visual shot embeddings where the configured model supports them.

Results hydrate to the representation passage, keyframe/crop, and raw locator. Text, media-vector,
and structured candidates can fuse with the existing RRF operator. This remains zero-generative-LLM
on the query path: it is embedding retrieval over a rebuildable P1 projection.

The model capability and index version must be explicit, just as D63 makes the text embedder a
versioned port configuration. A deployment without a compatible media embedder should advertise the
missing channel and return D49's typed `boundary`, not silently pretend caption search is equivalent.
The evaluation must separate text-to-image, image-to-image, text-to-acoustic-event, and transcript
retrieval; they are different tasks.

## 8. Mounts, P3 placement, cost, and requirement wording

### 8.1 Mount and serving refinements

The four-mount model in D51 should stand. Two refinements are needed:

- Raw media expected to be inspected must use a storage class and serving path compatible with its
  actual read/seek pattern. “Media → standard/nearline” is a starting heuristic, not enough by
  itself. Large-file gcsfuse behavior, nearline retrieval/minimum-duration charges, codec seeking,
  and API-side ranged access need measurement per MIME/size class.
- Full API/CLI parity should include typed source-locator access, not merely whole-object byte fetch.
  Otherwise mounted S59 can seek locally while unmounted S59 must download a multi-gigabyte file.

Derived previews and keyframes may be standard storage because they are on the navigation path.
They remain caches/artifacts and can be regenerated. Storage policy should deduplicate identical
derived assets by hash where practical and delete every representation child during hard-forget.

### 8.2 P3 placement

P3 should continue to materialize stubs, never raw media files. Standalone media lineages belong in
configured stable facets such as `by-type/images`, `by-type/audio`, and `by-type/video`, and may also
appear through topic, source, time, and entity views by stub duplication. Their canonical stable leaf
is still lineage-anchored per D55.

A media stub/member-table row should include modality, duration or dimensions, capture/publish date,
PageIndex root summary, current representation status, and conspicuous conversion warnings. It
points first to `document.md`, then explicitly to raw. Unsupported or failed conversion should still
yield a stub and raw pointer with `derived_representation: unavailable`; failure must not erase the
input from navigation.

Embedded/extracted figures stay under the parent artifact and are reached from its Markdown, as S56
already requires. Promoting every keyframe to a P3 document would explode the tree and falsely imply
independent source identity.

### 8.3 Cost and scale

Media changes the cost envelope more than ordinary documents:

- full-duration ASR, diarization, video decoding, shot detection, OCR, VLM calls, and media
  embeddings are separate metered components;
- keyframes and crops can multiply object count and storage;
- a long recording can produce far more blocks and claims than a typical PDF;
- remote model providers may require uploading sensitive raw bytes, a material deployment-policy
  choice even within one UGM trust domain.

Use the existing D52 ledger discipline per component and route. Video coverage must be bounded by an
explicit adaptive policy—at least full timeline shot coverage with recorded sampling inside shots—
and the manifest must disclose gaps. “One frame every N seconds” without coverage metadata is silent
loss. Audio speech coverage should span the full selected tracks; corrupt/unsupported intervals are
recorded as gaps rather than omitted.

## 9. Failure modes of this recommendation

This model is not lossless and should say where it can fail.

1. **Fluent derivation becomes false evidence.** An ASR substitution, hallucinated VLM object, or
   wrong speaker mapping can yield perfectly grounded text—in `document.md`—that the raw does not
   support. Mediation labels and raw anchors make this visible but do not make it correct. Raw-based
   multimodal audits and D22 canaries are mandatory, and high-impact model-observation claims may
   need stronger K3 admission policy.
2. **Diarization is not identity.** Stable `Speaker 2` segmentation can be accurate while mapping it
   to Bob is wrong. Conservative unresolved labels reduce useful attributed claims, but aggressive
   naming corrupts stance memory. The recommendation chooses the conservative failure.
3. **Video sampling misses transient content.** Shot keyframes can miss a one-frame slide, cursor
   action, gesture, or small changing value. Adaptive sampling and coverage reports reduce but do
   not eliminate this. Raw access remains the only lossless path.
4. **Representation generations increase state.** Keeping immutable old Markdown/source maps for
   each ASR/VLM upgrade costs storage and adds schema joins. Overwriting in place is cheaper but
   destroys historical grounding; I consider the extra generation identity non-negotiable.
5. **Markdown can become huge and noisy.** Long recordings with timestamps, overlapping speech,
   OCR, and shot notes can swamp PageIndex and E2. Semantic sectioning, role filters, and extraction
   batching help, but quality and cost must be measured by duration class. There must be no silent
   pre-extraction omission disguised as “summary.”
6. **Derived-text and direct-media search can disagree.** A caption may say “blue car” while the
   visual embedding retrieves it for “black vehicle.” RRF should return both with channel labels;
   it must not synthesize agreement. The agent audits raw when the distinction matters.
7. **Source deep links are codec- and client-dependent.** A logical time locator is portable, but
   seeking a remote variable-frame-rate video may require decoding from an earlier keyframe. The
   API can honor semantics only with codec-aware serving; a bare object-store byte-range promise is
   false.
8. **Model-family correlation is hidden by lineage counts.** Ten independent images captioned by
   the same VLM are ten source lineages but share one systematic perception error. D54's count is
   still the correct source count, yet it is not a calibrated probability. The envelope must retain
   derivation-family provenance so confidence policies can see the correlation.
9. **Metadata can be sensitive or misleading.** EXIF location, embedded captions, and participant
   lists may be useful and are already visible in raw under D51's one-trust-domain rule. Extracting
   them makes them easier to discover. Route manifests and deployment/provider policy must make this
   explicit; the library must not assume a cloud UI or an external policy authority.
10. **Exact reuse becomes too conservative.** Including source-map and mediation semantics in the
    representation key can reprocess text whose wording did not change. Excluding them can retain
    wrong speaker/time provenance. Correct grounding should win; spikes should optimize the hash
    granularity without weakening it.

## 10. Open spikes

These are measurements needed to choose parameters and adapters, not reasons to leave the conceptual
model incomplete:

1. **Route quality corpus.** Build a multilingual, noisy media golden set and measure ASR word error
   rate, diarization error rate, speaker-attribution precision, OCR accuracy, VLM factuality, and
   time/region alignment separately. Include overlapping speech, screen recordings, charts, music,
   and corrupt tracks.
2. **Grounding precision.** Measure how often word timestamps versus segment timestamps let a claim
   open a short enough interval to audit, and how often image/video claims need multiple regions or
   times.
3. **Raw-based audit policy.** Determine modality-stratified sampling rates and escalation bands for
   D32's independent audit. Compare auditing derived text alone with auditing the referenced raw
   interval/region; only the latter tests converter fidelity.
4. **Representation lifecycle.** Exercise ASR/VLM upgrades that (a) change text, (b) change only
   timestamps/speakers, and (c) yield identical outputs. Verify atomic basis swaps, claim reuse,
   `support_withdrawn`, `claims_as_of`, and K stale triggers.
5. **Transcript/video structure.** Evaluate PageIndex topic boundaries and block/chunk stability over
   long recordings. Measure whether deterministic scene/chapter headings improve retrieval without
   turning every speaker turn into a section.
6. **Visual description granularity.** Compare whole-image captions, region proposals, OCR-first
   prompting, and chart-specific adapters. Optimize claim recall and hallucination rate, not prose
   richness.
7. **Video coverage policy.** Measure shot detection plus adaptive within-shot sampling against
   transient-event recall, derived object count, VLM cost, and multimodal-index size. Bind coverage
   thresholds by media class from these results.
8. **Direct media search.** Evaluate text-to-image, image-to-image, text-to-acoustic-event, and
   text-to-video-shot recall independently. Compare added recall over OCR/ASR/VLM text and size the
   `media_segments` P1 target accordingly.
9. **Seek and parity.** Test gcsfuse reads, codec seeking, locator-aware API streaming, and storage
   class charges on representative large files. S59 must pass without forcing an unmounted agent to
   download an entire multi-gigabyte video merely to inspect ten seconds.
10. **Cost and retention.** Measure per-hour ASR/diarization/VLM/embedding spend, keyframe and
    sidecar storage, representation-generation growth, and hard-forget latency at the million-document
    target.
11. **Provider/privacy routes.** Verify which adapters can run locally versus sending raw media to a
    provider, and ensure the derivation manifest records that execution context. This is deployment
    policy through D61 provider ports, not an external correctness mechanism.
12. **Agent comprehension.** Extend S58: give a cold harness mixed OCR, ASR, and VLM claims and test
    whether the consumption skill makes it distinguish source expression, model observation, media
    time, asserted world time, and current fact state.

## 11. Proposed decision set

The minimal coherent decision package is:

1. Media is an E0 input modality, not a new evidence plane. Every input gets a Markdown-first
   document representation and retains one explicitly reachable immutable raw asset.
2. `document.md` contains all text eligible for chunking/extraction. `media/` contains only derived
   support assets; whole-file originals remain raw.
3. The converter contract becomes `document.md + source_map + derived_assets + manifest`, with bound
   standalone image, audio, and video routes.
4. Source provenance is a typed page/region/time locator chain from claim span to an immutable raw
   document version. Evidence hydration and P3/raw pointers expose it on mounted and unmounted
   surfaces.
5. Claims disclose whether their support is source expression, model observation, or model
   interpretation, plus the derivation run. D32 audits the raw modality, not only the generated text.
6. A document version owns immutable representation generations. Representation generation joins
   D54's extraction basis; converter upgrades are processing-driven re-transcriptions and use
   `support_withdrawn`, never source-removal retraction.
7. P1 retains text search as the default and adds a locator-bearing `media_segments` semantic target
   for visual/acoustic discovery that descriptions cannot cover.
8. P3 holds media document stubs and derived previews, never whole raw media or per-keyframe pseudo-
   documents. Raw remains off-path but fully reachable.

That package implements the owner's requirement without weakening Markdown-first navigation, raw
auditability, lineage counting, rebuildability, or the library boundary.
