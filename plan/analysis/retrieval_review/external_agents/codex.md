# Codex review — retrieval design PR (design/retrieval, July 2026)

Reviewer: Codex (gpt-5.5), adversarial-review prompt over the full binding corpus.
Reconciliation: all 17 findings accepted (some with narrower fixes); applied in the same PR.

1. [SEVERITY: blocker] [plan/designs/retrieval_design.md §2/§7] — D48 says every result that reaches a caller passes live Postgres hydration and that stale projections cannot affect correctness, but D51 then makes mounted K/P3/E0 filesystem reads the preferred path; those reads bypass API hydration entirely. P3/K snapshots can be stale and raw/artifact reads carry no live validity check, so the invariant is false for a primary consumption surface. — Proposed fix: scope D48 to API/CLI query-engine responses, and require mounted surfaces to expose snapshot/freshness metadata while the consumption skill requires spine verification before treating mount content as current belief.
2. [SEVERITY: blocker] [plan/designs/retrieval_design.md §2/§5] — K pages are treated as lagging projections whose candidates can be "disposed" by PG hydration, but compiled prose and authored content have no row-level validity windows that Postgres can re-read. Hydrating cited IDs can reveal staleness; it cannot make a stale synthesis correct or drop only the superseded sentence. — Proposed fix: classify K answers strictly as compiled grain with staleness/open-flag status, never as live-confirmed truth; route load-bearing current-belief checks through explicit relation/observation lookups.
3. [SEVERITY: blocker] [plan/designs/retrieval_design.md §11; plan/analysis/retrieval_scenarios.md S55; questions.md §4 #24] — S55 and retrieval CI require forgotten ≡ never-existed across semantic, verbatim, graph, K, and browse paths, but the open questions still say P1/Lance, immutable P2/P3 snapshots, PITR/backups, and the full cascade are unresolved. The retrieval design cannot make an unresolved lifecycle problem a non-negotiable query contract. — Proposed fix: either design the end-to-end purge/snapshot-retention behavior now, or mark S55 as an open boundary and remove it from hard retrieval CI until deletion is closed.
4. [SEVERITY: major] [plan/designs/retrieval_design.md §3; plan/designs/p2_graph_design.md §8] — Every primitive accepts `believed_at`, and D49 says transaction-time caps apply coherently across all channels, but P2's hot snapshot only keeps invalidated edges within a retention window and P1/K historical snapshot behavior is unspecified. Old audit-date graph/path queries will either lose retracted edges or need a fallback the design does not define. — Proposed fix: expose the transaction-time horizon per channel and define fallback to archived snapshots or PG relation traversal; otherwise return a typed `boundary` for out-of-retention `believed_at`.
5. [SEVERITY: major] [plan/designs/retrieval_design.md §2/§3] — Hydration verifies individual IDs, but graph paths are only correct as whole paths. If stale P2 nominates a shortest path and hydration drops one edge, the remaining records are not a valid path; dropping rows is not enough to preserve correctness. — Proposed fix: require path-level revalidation after live filtering, drop invalid paths as units, and state whether alternate live paths are recomputed or only recall-loss/truncation is reported.
6. [SEVERITY: major] [plan/designs/retrieval_design.md §5/§13] — The envelope says returning one side of a live contradiction without all co-members is a contract violation, but §13 later contemplates inline co-member caps with truncation if groups are large. Those two rules conflict, and hub answers can duplicate large co-member sets per result. — Proposed fix: make contradiction groups a bounded subresource: inline only under a guaranteed cap; otherwise return `group_id`, returned count, total/estimated total, and continuation.
7. [SEVERITY: major] [plan/designs/retrieval_design.md §5/§6] — The envelope has one top-level `grain`, but S47 requires a single response containing an evidence timeline and a belief snapshot. A single grain field cannot be truthful for mixed-grain responses unless a mixed type is added, which §6 explicitly avoids. — Proposed fix: model compound answers as `parts[]`, each with its own grain and envelope metadata, or require callers/recipes to return separate envelopes.
8. [SEVERITY: major] [plan/designs/retrieval_design.md §4/§6; plan/designs/postgres_schema_design.md §16] — Recipes are binding registry data, but no schema or DSL exists for recipe rows, primitive chains, parameter typing, grain inference, aliases, versioning, or MCP rendering. The proposed linter also relies on detecting whether a name/description "implies current-belief," which is semantic policy rather than a mechanical guard. — Proposed fix: define a `retrieval_recipes` control-plane schema with explicit `output_grain`, `answer_intent`, allowed input grains, typed primitive DAG, and MCP metadata; make prose-name lint advisory or constrained by enums.
9. [SEVERITY: major] [plan/designs/retrieval_design.md §3; plan/designs/registries_design.md §5; plan/analysis/retrieval_scenarios.md S50] — `resolve` is query-time T0-T2 only, while D17's authoritative resolution cascade relies on T3 embeddings and T4 adjudication for ambiguous residue, and Czech canonicalization is produced during E2 extraction, not for raw user queries. S50 also uses the nominative "Jiří Puc," so it does not actually test inflection. — Proposed fix: define query `resolve` as T0-T3 non-LLM candidate generation with ambiguity returned, add a query-string normalization story if needed, and replace S50 with an inflected form such as "Jiřího Puce."
10. [SEVERITY: major] [plan/designs/retrieval_design.md §3/§9; plan/analysis/retrieval_scenarios.md S30/S40] — `aggregate` is limited to count, group-by-predicate, group-by-object, and timeline(entity), but the scenario battery expects "entities gained the most new evidence this month" and "modules with no documented interfaces." Those require group-by-entity over deltas and anti-join over typed entity enumeration, neither of which is enumerated. — Proposed fix: add those bounded SQL shapes explicitly, or reclassify S30/S40 as batch/boundary scenarios.
11. [SEVERITY: major] [plan/designs/retrieval_design.md §3; decisions.md D21/D44] — `believed_at` composition ignores entity-resolution history and merge redirects. A current `resolve`/hydrate path will follow today's aliases and `merged_into` survivor chain, which can answer an audit-date query with identities the system did not believe at that date. — Proposed fix: specify as-of entity resolution over `resolution_decisions`/`merge_events` and survivor redirects at `believed_at`, plus scenarios for merged aliases and un-merge.
12. [SEVERITY: major] [plan/designs/retrieval_design.md §7/§9] — §7 says "the API is the one place authorization is enforced for query-engine reads," but §9/D50 says content-level authorization and per-user scoping are library non-goals and perimeter security is deployment infrastructure. Implementers are left unsure whether API authz is a library contract or an ops concern. — Proposed fix: use one vocabulary: API/mounts may authenticate/admit callers at the deployment perimeter; the library must not content-filter inside a deployment.
13. [SEVERITY: major] [plan/requirements/requirements_v3.md §Plane P; plan/designs/overall_design.md §5; plan/designs/e0_files_design.md §6] — Requirements and overall still say P3 is built from "entities/relations + the K-plane structure," while E0 §6 and schema §17 say K is only cross-linked and is not a structural input. This changes rebuildability, deletion reach, and whether P3 inherits K's git source-of-truth burden. — Proposed fix: pick the E0 model and update D40/requirements/overall, or redesign P3 as K-dependent with explicit deletion and regeneration rules.
14. [SEVERITY: major] [plan/analysis/retrieval_scenarios.md §Coverage; plan/designs/retrieval_design.md §11] — The battery claims to cover the query surface, but it has no scenarios for entity merge/unmerge, stale aliases, resolver transcript retrieval, or graph endpoints redirected by D44. Those are central to D21/D44 and can change every retrieval channel's answer. — Proposed fix: add scenarios for old names after merge, audit-date pre-merge resolution, un-merge reversal, and P2 survivor-redirect correctness.
15. [SEVERITY: minor] [plan/designs/retrieval_design.md §3-§5; plan/analysis/retrieval_scenarios.md §M] — Rule 1 issue: important retrieval terms are named faster than they are explained for cold readers, including BM25, FTS, RRF's ranking behavior, cross-encoder, Daitch-Mokotoff, MCP, and continuation-token mechanics. Some are expanded, but not enough to explain what problem they solve and why they are chosen. — Proposed fix: add a short retrieval glossary or expand first uses with plain-language examples.
16. [SEVERITY: minor] [plan/analysis/retrieval_scenarios.md title/intro/references] — The scenario doc still says "S1-S55" and describes `retrieval_design.md` as "(planned)" even though S56-S59 exist and the design is now binding. This is stale metadata in the document that is supposed to drive D22 eval. — Proposed fix: rename the scope to S1-S59 and remove the planned-language references.
17. [SEVERITY: major] [decisions.md D42] — Rule 2 leakage remains in the decision log: "deliberately deferred," "documented non-goal for now," and "other deferred items remain non-goals" appear in a binding decision. Even if D45-D47 refined part of D42, the remaining phrasing reads as phasing rather than full-scope design. — Proposed fix: rewrite those as scope boundaries or documented alternatives without "for now/deferred" language.

Overall verdict: Not ready to bind.
Core retrieval direction is sound, but D48 overclaims live correctness across K and filesystem surfaces.
Hard-delete and transaction-time semantics are not closed enough for CI contracts.
Recipe/envelope mechanics need concrete schemas before implementation.
Accept after tightening authority boundaries, temporal/identity semantics, and scenario coverage.
