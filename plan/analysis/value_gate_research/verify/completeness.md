## gaps[]

**G1 — The gate's own cost is asserted cheap, never bounded against the thing it gates with a worked floor.** The task's pressure-test (a): *does the gate risk being a new expensive LLM stage?* The corpus answers this **directionally but not rigorously**. V2 §2.2 and V5 rec-4 and V6 §2.4 all say "gate ≪ extraction" and "the LLM rung piggybacks E1's cached context call." But: (i) **no doc quantifies `g` (gate cost per section) as a number or even a hard ceiling** — V6's cost model carries `g` as a symbol and then drops it ("+ g") without bounding it; (ii) the "piggyback E1's cached context call" claim is an **architectural assumption that is never validated against the actual E1 design** — if the salience rung needs a *different* prompt than E1's context-prefix prompt, the prompt cache does not help and it becomes a genuine extra LLM call per section. **At 1M docs × N sections, even a "small-model" call per section is a new fleet-scale LLM stage.** None of V2/V5/V6 models the gate's *aggregate* cost at corpus scale, only its per-item ratio. This is the single biggest under-answered risk and maps exactly to the task's concern (a).

**G2 — Section count is unmodeled; the gate's scale is hidden.** V6 repeatedly says "per-section, document-rollup" and asserts `gate_decisions` rows are "≪ chunk count." But **no doc estimates sections-per-document** for a 1M-doc corpus. If PageIndex yields 20–50 nodes/doc, that is 2–5×10⁷ gate decisions, each potentially a small-LLM call (G1). The "negligible table" claim (V6 §2.6) is plausible for storage but **says nothing about the compute** of producing those rows. Gap: sections/doc × gate-cost/section is the real number, and it is absent everywhere.

**G3 — Rebuildability of gate decisions: the hard case (non-determinism) is dodged.** Task pressure-test (b). V3/V6 do well on the *schema* (versioned `gate_decisions`/`document_extraction_state` rows, queue-as-projection, outbox). But they assert the verdict is "deterministically derivable" (V3 §2.3) **while simultaneously specifying a small-LLM salience rung** (V2 rung 3/4, V6 `T-salience`). A small-LLM score is **not deterministic** across model/temperature/version. So "re-run the gate → identical verdict" is false in general; rebuild reproduces only "the corpus as gated *under that gate_version*," and a rebuild after a model deprecation may produce *different* verdicts for the same `gate_version` label if the underlying model endpoint drifted. None of the docs address **model-endpoint drift vs. version-pinning** — the exact "rater drift" risk V5 §2.4 raises for thresholds but never carries through to rebuildability. The deterministic rungs (hash, structural, near-dup) ARE replayable; the salience rung is only replayable if its output is *stored*, not *recomputed*. V6's schema stores `features jsonb` and `tier` (good — replayable by storage) but the prose still claims "recomputable" (misleading). **Gap: the docs conflate "stored and auditable" with "deterministically recomputable"; only the former is true for the LLM rung.**

**G4 — Recall safeguards lean on triggers that are themselves unvalidated, and one safeguard has a circular dependency.** Task pressure-test (c). V5 + V3 build the safeguard set on: always-full-E1, retrieval-triggered backfill, scope-interest backfill, bounded drain, canaries. Problems left open:
  - **Retrieval-triggered backfill has a cold-start hole the docs name but under-weight.** "You cannot retrieve what was never indexed" is handled by always-full-E1 (chunks searchable). But a deferred doc's chunks compete in P1 *without* the claims/relations boost — so a low-salience-but-critical fact in a deferred doc is **less retrievable than the same fact in a full-extracted doc**, precisely when it most needs to fire its own backfill trigger. V3 §2.5 mitigation-1 asserts retrievability; it does **not** show the deferred chunk actually ranks high enough to be retrieved and self-promote. The penicillin-allergy gem could sit in a deferred doc, never rank top-k (no relation edges, no evidence_count), and never self-trigger. **The safeguard is necessary but not shown sufficient.**
  - **`extract-on-scope-interest` (D16) is flagged by V2 §3 and V3 §3 as having NO external prior art and being unvalidated** — yet V3/V5/V6 promote it to the *highest-leverage* mitigation. A load-bearing safeguard rests on an unbuilt, unmeasured mechanism. That is a coherence gap: the recall story's strongest leg is its least-evidenced.
  - **Never-defer classes depend on D5/D15 ontology that flags medical/legal predicates — but predicate typing happens *during/after* extraction (E2/E3), and the gate runs *before* E2.** V5 rec-3 says "never-defer type/predicate classes bypass the gate to EXTRACT_NOW." But the gate cannot know a section contains a penicillin-allergy *predicate* without having extracted it. The only pre-extraction signal is **lexical** (keyword/section-type), which is far weaker than the predicate-registry framing implies. **Circularity: the safeguard that protects rare medical facts needs the extraction it is gating.** No doc resolves this; V5 gestures at "up-weight temporal/change-of-state language" (V6 §3) but admits the heuristic is "unvalidated."

**G5 — O3's "plausibly 10×" is repeatedly de-claimed but never replaced with a committed planning number.** Every doc (V1, V3, V4, V6) correctly says salience-skip alone is ~1.5–2× and 10× needs the lazy tier carrying most weight. Good. **But the synthesis-level question — "what cost multiplier should ugm actually plan to?" — is left as "measure it" (spike-1) in all six docs.** That is honest, but it means O3's headline benefit is, after six research docs, still unquantified for ugm. A completeness gap: the research validates the *mechanism* and *ceiling* but delivers no defensible *expected* number, and does not state what minimum multiplier would make the gate worth its own complexity/risk (the break-even is never computed).

**G6 — The "zombie fact" / supersession-skip risk is identified but its mitigation is hand-waved.** V6 §3 names the existential case: if a `chunks_only`/`deferred` section held the *only* superseding evidence, skipping it serves a stale fact as current. The proposed fix — "up-weight temporal/change-of-state language toward full" — is explicitly "unvalidated" (V6) and appears in **only one doc** (V6); V5's recall treatment does not integrate it. This is the highest-severity correctness risk of the whole gate (it can make the system *confidently wrong*, not just incomplete) and it gets one unvalidated sentence. Under-answered relative to its severity.

**G7 — No doc addresses gate-vs-extraction *latency/throughput coupling* operationally.** V6 §2.5 flags first-retrieval latency for deferred structured facts as "needs a product decision" — but nobody addresses the *ingest-time* consequence: inserting a new mandatory synchronous stage (E1.5) into the per-doc Cloud Tasks chain adds a hop, a queue, a failure mode, and 2-retries+DLQ surface for *every* document, including the ones that end up `full` anyway (who now pay gate cost *plus* full extraction). For the `full` fraction, the gate is **pure overhead**. No doc nets the gate cost against the `full`-tier docs that gain nothing from it.

**G8 — O6 dependency is correctly flagged everywhere but never scoped.** All six docs end with "untunable without O6's golden set." None estimates the **labeling cost** of the gate-verdict golden set (sections labeled should-full/fine-to-skip + planted canaries) or who produces it. The gate is gated on an unscoped prerequisite. (Lower severity — correctly out of scope per-doc — but unowned across the set.)

## contradictions[]

**C1 — [MAJOR] The "98% junk" claim: V1 says VERIFIED & traceable; V4 and V6 say UNVERIFIED / no source exists.** This is a direct factual contradiction within the same research set.
  - V1 §1/§2.1: *"The '98% junk' claim is REAL and traceable … mem0 GitHub issue #4573 … 97.8% … [VERIFIED — issue fetched twice]."*
  - V4 §3: *"my web search for a mem0 LOCOMO junk-percentage returned no source for it … Treat the 98% as an unverified rhetorical figure."*
  - V6 §3 + rec-6(d): *"the '~98% junk' figure in O3 is NOT independently verified … Drop the unverified '98% junk' anecdote from planning inputs."*
  - **Adjudication [CHECKED]: V1 is correct.** Issue #4573 exists with exactly that title and number. V4/V6 searched for the wrong artifact (a "mem0 LOCOMO junk-rate," which indeed does not exist) and concluded the claim was unsourced. **V4 and V6 are wrong on this point and their rec-6(d) "drop the anecdote" advice is mis-grounded.** The *correct* nuance is V1's: the claim is real but n=1, pathological setup (52.7% = an agent re-ingesting its own boot file), not a population statistic. Any synthesis that adopts V6's "drop it" guidance over V1's "demote it, fix the citation" guidance would be acting on a falsified premise.

**C2 — [MODERATE] Gate granularity: V2 says per-section *with per-chunk overrides*; V6 says per-section is primary and *per-chunk is too fine / only a fallback*; V5 frames the gate decision at the *document/section row* level.** V2 §4 ("runs per section … with per-chunk overrides"), V6 §2.1 ("per-chunk is too fine to be the primary unit … fallback only"), V5 rec-1 ("on the document/section row"). These are reconcilable but **not reconciled** — the three docs never agree on the canonical unit, and the cost model (V6) and the safeguard model (V5) are computed at *different* granularities, which means their cost/recall numbers are not directly composable. A synthesizer must pick one unit; the docs don't.

**C3 — [MODERATE] Is the defer/gate decision "deterministic"? V3 says YES (deterministically derivable, rebuildable); V2/V6 specify a small-LLM salience rung (non-deterministic).** V3 §2.3 leans the entire D7-safety argument on "the defer verdict is exactly such a derivable artifact." V2 rung-3/4 and V6 `T-salience` introduce an LLM scorer. An LLM verdict is not deterministically *re-derivable* (see G3). The docs paper over this by storing features+verdict, but the *prose* claims contradict each other on whether the verdict is recomputed or merely replayed-from-storage.

**C4 — [MINOR] mem0's recall-bias stance: V4/repo_findings present "when in doubt, extract" as an *anti-pattern* ugm should invert; V1 and V5 partially *adopt* it (recall-biased gate, "tilt conservative / extract slightly too much").** Not a hard contradiction (V1/V5 apply recall-bias to the *cheap* decision, mem0 to the *paid* one), but the framing whiplash — mem0's maxim is simultaneously "the bug" (V4) and "the right default" (V5 guardrail, V4's own guardrail section) — is unreconciled and could confuse a synthesis into both keeping and discarding the same principle.

**C5 — [MINOR] LazyGraphRAG as "the lazy ceiling proves 10× is in reach" (V6) vs. "LazyGraphRAG is NOT a reusable architecture for ugm" (V3).** V6 §2.6 uses LazyGraphRAG's 0.1% as evidence the 10× ceiling "exists, so 10× is conservatively in-reach." V3 §1 stresses LazyGraphRAG defers *100%* and *re-pays every query / persists nothing* — the **opposite** trade from ugm. Using its 0.1% indexing figure to underwrite ugm's expected savings (V6) is exactly the apples-to-oranges V3 warns against. Both docs are individually careful, but their conclusions pull opposite directions on how much weight LazyGraphRAG's number should carry for ugm.

## overclaims[]

**O1 — "The gate is reversible / low-regret almost for free" (V5 §1, V6 §2.3) overclaims the LLM rung's reversibility.** True for the deterministic rungs and true that *input* is never deleted (E0/E1 immutable). But "almost for free" hides: (a) the gate's own aggregate compute (G1/G2), (b) the non-determinism of re-deriving LLM verdicts (G3/C3), and (c) the recall holes in the trigger mechanisms (G4). "Reversible" is accurate at the *data* layer (nothing deleted) and overclaimed at the *decision* layer (an LLM verdict is not freely reproducible).

**O2 — Cost-savings bands in V6 §2.6 are presented in a table with specific percentages (65%, 55%, 36%) that look measured but are admittedly invented.** V6 labels them "modeled, not measured" in §3 — honestly — but the §2.6 table format (clean filter-rate rows, a "≈ 36%" column) reads as analysis output and is the kind of number that survives into a synthesis stripped of its caveat. The `f_full/f_def/f_chunk/r_retrieve` inputs are **pure assumption**; the table has no empirical anchor for *any* cell except the external LazyGraphRAG row. Risk: the bands get quoted as ugm projections (the very thing V3 §3 and V6 §3 warn against).

**O3 — "LazyGraphRAG re-extracts per query and does not cache" stated with more confidence than the source supports.** V3 honestly flags it as inference; but V4 §2.3 and V6 §2.2 carry it forward as a settled property to argue "ugm improves on LazyGraphRAG by persisting." [CHECKED] the blog is **silent** on caching. The "ugm beats LazyGraphRAG by persisting once" positioning rests on an *unconfirmed assumption about a competitor's internals*. Defensible as a design contrast; overclaimed as a factual gap in LazyGraphRAG.

**O4 — Denoising-KG "+8 to +17pp" and "40% entities removable" treated as a near-universal license to gate aggressively (V1 rec-4, V5).** The paper studies *post-hoc KG denoising* (prune an already-built graph), not *pre-extraction value gating* (decide what to extract). V5 §2.2 honestly flags the analogous Zero-RAG caveat ("they prune what the model already masters") but V1 rec-4 stretches the denoising result to "the safe operating point is well into aggressive territory" — a transfer from *graph-pruning-improves-retrieval* to *skipping-extraction-is-safe* that the paper does not license. The two operations have different failure modes (pruning keeps the option to re-add from a built graph; skipping never builds it). Directionally supportive, over-generalized as a green light for aggression.

**O5 — "Embedding-novelty is marginal-zero cost because E1 already embeds" (V2 §2.1, V6 §2.4) understates the lookup cost.** The *embedding* is sunk, yes. But the **novelty check is a top-k similarity query against the existing claim/relation corpus** for every candidate section — at 10⁸ existing claims that is a non-trivial vector-search load per section, not "marginal-zero." V2 calls it "a cosine lookup," eliding that it's a corpus-scale ANN query, not a single cosine. Cost is *lower* than an LLM call, but "marginal-zero" is an overclaim at 1M-doc scale.

**O6 — "No prior system implements a value gate" is robustly supported, but the adjacent claim that this makes it a clean 'differentiator' overclaims by omission.** The repo archaeology (grep-absence across 5 repos) is solid [verified in repo_findings]. But "unbuilt = differentiator" ignores that **the reason nobody built it may be that the recall risk (G4/G6) is not worth the cost saving** — the absence is presented as opportunity, never as possible evidence that the trade is unfavorable. mem0 #4573's own community *requested* a gate, which supports demand; but no surveyed system's *maintainers* shipped one despite knowing the cost, which the docs never treat as a signal.

## top-5-for-synthesis

1. **Resolve C1 in V1's favor: the "98% junk" claim IS real (mem0 #4573, 97.8%, [CHECKED]) but is n=1 and pathological (52.7% = agent eating its own boot file).** Adopt V1's "demote to a documented single-deployment anecdote, fix the citation" — **reject V6/V4's "drop it as unverified,"** which is grounded in a search for the wrong artifact. The load-bearing evidence for the gate is NOT the 98%; it is (a) the model-swap finding that a *better* model extracts *more* junk, (b) the denoising-KG causal lift, (c) corpus 5–10% survival. Lead with those.

2. **Close the gate-cost gap before committing (G1+G2+O5).** The corpus never bounds the gate's *aggregate* compute at 1M-doc × N-section scale, and the "piggybacks E1's cached call / marginal-zero novelty" claims are unvalidated architectural assumptions. **Mandate the spike to measure: sections/doc, gate-cost/section, and whether the salience rung can truly reuse E1's prompt cache.** If it cannot, the gate is a new fleet-scale LLM stage — the exact risk that motivates it. This is the #1 unaddressed risk.

3. **Fix the determinism/rebuildability story (G3+C3).** State plainly: deterministic rungs (hash/structural/near-dup) are *recomputable*; the LLM salience rung is *only replayable from stored verdict+features*, not deterministically re-derivable, and is subject to model-endpoint drift. The D7 guarantee for the gate is **"stored & auditable," not "recomputable"** for the LLM rung — pin model versions AND store outputs, don't promise recomputation.

4. **Harden the rare-fact safeguards — they are necessary but not shown sufficient (G4+G6).** Three holes: (i) deferred-doc chunks may never rank high enough in P1 to self-trigger their own backfill (the gem stays buried); (ii) the highest-leverage mitigation (extract-on-scope-interest) is unbuilt and unvalidated prior art; (iii) "never-defer medical/legal classes" is circular — predicate type is only known *after* extraction, so the gate has only weak lexical signal pre-extraction. The zombie-fact/supersession-skip risk (one unvalidated sentence in V6) is the highest-severity correctness failure and needs real treatment, not a heuristic.

5. **Commit a break-even, not just "measure it" (G5).** Six docs correctly de-claim O3's "10×" (salience-skip ≈1.5–2×; 10× needs the lazy tier) but none states the *minimum* cost multiplier at which the gate's added complexity, latency hop (G7), recall risk (G4/G6), and own compute (G1) are worth it. Synthesis should define that break-even and gate the whole feature on the spike clearing it — otherwise the gate is justified by mechanism plausibility, never by a net-positive number.
