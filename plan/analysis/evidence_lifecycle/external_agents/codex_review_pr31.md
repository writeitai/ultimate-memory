# Codex review — PR #31 (evidence lifecycle + E1 chunking), July 2026

Reviewer: Codex (gpt-5.5), adversarial prompt over the full branch + binding corpus.
Reconciliation (final say: Claude): 15/16 accepted fully; F9 accepted as a D31 refinement (batching preserves the two-call shape + per-chunk ledger commits — no redesign); F16 accepted as rewording (documented-alternative framing is repo-legal; recast, not removed). All fixes applied in the same PR.

1. [SEVERITY: blocker] [plan/designs/postgres_schema_design.md §1/§6] — The DDL references `document_origin` in `documents.origin`, but §1 never creates that enum. This makes the schema text non-executable before any lifecycle behavior is reached. — Add `CREATE TYPE document_origin AS ENUM ('external','system_generated', ...)` in §1, or replace the column with an existing type and update D42 references consistently.

2. [SEVERITY: blocker] [plan/designs/postgres_schema_design.md §6/§10.A; plan/designs/p2_graph_design.md §2] — `published_at` moved to `document_versions`, but `v_graph_documents` selects `published_at` from `documents`; the graph design correctly says it should use the current version. The view as written cannot compile and the P2 Document node loses its current-version date semantics. — Rewrite `v_graph_documents` to join `documents.current_version_id` to `document_versions.version_id`, select `dv.published_at`, and handle NULL/current-version-missing cases explicitly.

3. [SEVERITY: blocker] [plan/designs/evidence_lifecycle_design.md §2; plan/designs/postgres_schema_design.md §6/§8/§9] — `removal_semantics` is binding in D55 and lifecycle §2, but the schema has no column or enum for it, and `removed_from_source` is not representable in `currency_reason`, `adjudication_outcome`, or a dedicated invalidation-reason field. The advertised `review` vs `retract` behavior cannot be persisted or audited. — Add a per-lineage `removal_semantics` enum column, add an explicit adjudication/invalidation reason that includes `removed_from_source`, and state which relation/observation adjudication rows are written for mechanical retraction.

4. [SEVERITY: blocker] [plan/designs/evidence_lifecycle_design.md §6; plan/designs/postgres_schema_design.md §7/§8] — Claim reuse across versions is not representable: `claims.chunk_id` points to one chunk row, while the design says a new version's chunk row "points at" reused claims and chunk rows double as occurrence records. Matching only by `chunk_content_hash` cannot distinguish duplicate identical chunks in one lineage/version and cannot attach one immutable claim to multiple version occurrences. — Add a `claim_occurrences`/`chunk_claims` table keyed by `(claim_id, version_id, chunk_id)` or introduce a stable occurrence identity; make `claims_as_of`, currency transitions, and K citation keys read that occurrence table.

5. [SEVERITY: blocker] [plan/designs/postgres_schema_design.md §9] — `relation_evidence` is declared `PARTITION BY HASH (relation_id)` but, unlike `observation_evidence`, no hash child partitions are created. Inserts into the table will fail unless migrations add partitions out-of-band, contradicting this document's claim to specify the complete schema. — Add the same static partition creation block used for `observation_evidence`, or state a mandatory Alembic helper that creates the partitions.

6. [SEVERITY: major] [plan/designs/postgres_schema_design.md §6] — `documents.current_version_id` only references `(deployment_id, version_id)`, so a document can point at another lineage's version in the same deployment. That breaks testimony currency, P2 current-version projection, and any "current source statement" read. — Add a composite uniqueness target on `document_versions(deployment_id, doc_id, version_id)` and reference `(deployment_id, doc_id, current_version_id)` from `documents`.

7. [SEVERITY: major] [plan/designs/evidence_lifecycle_design.md §4/§5; plan/designs/postgres_schema_design.md §9/§13] — The recount path still contains old claim-row semantics: §13 says recompute counts with `COUNT(DISTINCT claim_id)`, directly contradicting D54's "distinct current-testimony lineages" denominator. The schema also lacks a lineage-indexed support table, so reconciliation's "bounded and indexed" lineage diff becomes claim lookup plus all-partition evidence reverse scans. — Define the canonical recount SQL as `COUNT(DISTINCT claims.doc_id)` filtered by current testimony and stance, and either denormalize `doc_id`/currentness into support rows or add a materialized `fact_lineage_support` table maintained by reconciliation.

8. [SEVERITY: major] [plan/designs/evidence_lifecycle_design.md §5; plan/designs/postgres_schema_design.md §6/§8] — The `retract` move rule depends on "after the connector's sync cycle completes", but no sync-cycle id, barrier state, or transaction boundary exists in the schema. Concurrent per-lineage completions can still retract sole support before the destination lineage's moved content finishes extraction. — Introduce a connector sync-cycle table and make retraction evaluation a cycle-finalization job over all completed lineages, with a grace/error state for incomplete extractions.

9. [SEVERITY: major] [plan/designs/e1_chunks_design.md §6; decisions.md D31; plan/designs/postgres_schema_design.md §2/§8] — E1 says E2 batches a section's contiguous chunks into one call, while D31 still binds extraction to two calls per target chunk unless an ablation proves collapse safe. The `processing_state` and `cost_ledger` keys are per target chunk/stage, so one call spanning N chunks has no idempotent target or cost anchor. — Either amend D31 to define batched Selection and batched decontext/decompose semantics, or add an extraction-batch artifact keyed by a `batch_input_hash` with per-chunk commit rows and cost allocation.

10. [SEVERITY: major] [plan/designs/e1_chunks_design.md §7; plan/designs/e2_e3_claims_relations_design.md §3.1; decisions.md D31/D56] — D31 says PageIndex section path/summary and the E1 prefix are part of the extraction bundle, but D56 excludes LLM-derived context from the reuse key and carries it forward. A structurer or role change can therefore leave old Selection/decontextualization decisions in force even though the binding E2 input would have changed. — Split extraction context into stable inputs that must key reuse and replayed context that is part of the old extraction record; force re-extraction when any context that can affect Selection changes, or explicitly bind reused claims to the old context and prevent new section roles from changing their interpretation.

11. [SEVERITY: major] [plan/designs/postgres_schema_design.md §8/§12] — `testimony_currency_events` can grow at claim scale under re-extraction and living-version churn, but it is not in the partitioning table and has no uniqueness/idempotency key for a transition. This violates the schema's own large-table discipline and leaves retries able to append duplicate currency events whose replay order is ambiguous. — Partition it by `occurred_at` or by `doc_id`, add an idempotency key such as `(claim_id, became_current, reason, basis_id/reconciliation_id)`, and include it in §12 and the logical-FK auditor rules.

12. [SEVERITY: major] [plan/designs/evidence_lifecycle_design.md §8; plan/designs/postgres_schema_design.md §13; plan/designs/retrieval_design.md §5] — Lifecycle §8 says deleting a version ends its claims' currency while the lineage continues, but schema §13 deletes chunks, claims, decisions, audits, and evidence rows during normal delete. That destroys assertion history and makes `claims_as_of` and audit opt-ins unable to distinguish deleted-version history from never-ingested content, while S55's "forgotten ≡ never existed" is supposed to apply only to hard-forget. — Separate normal version/lineage deletion from hard-forget: normal deletion should tombstone versions and close currency; hard-forget should scrub/delete source-bearing payloads and then satisfy S55.

13. [SEVERITY: major] [plan/designs/e1_chunks_design.md §2] — The blockizer determinism claim is underspecified for a load-bearing identity layer: it says "CommonMark" but also "pipe tables", which are not CommonMark, and does not pin parser profile, extension set, or canonicalization edge cases. Two compliant implementations could hash different block sequences. — Specify the exact parser/library profile and version, enabled extensions, normalization order, and a golden corpus whose block hashes are regression-tested per `blockizer_version`.

14. [SEVERITY: major] [plan/designs/e1_chunks_design.md §3] — The section snap rule is too vague for malformed PageIndex output: overlapping siblings, duplicate starts, gaps before/after children, child ranges outside parents, and zero-length sections all fall into "degrade to parent" without an algorithm. Implementers can produce different partitions, which then changes chunking and reuse. — Define a total normalization algorithm over sorted proposed spans, including root coverage, tie-breakers, nesting repair, gap assignment, zero-length pruning, and deterministic role/title inheritance.

15. [SEVERITY: major] [plan/designs/e1_chunks_design.md §4/§7] — Anchor-stabilized packing is named as load-bearing but not specified enough to implement: the hash criterion, anchor density target, interaction with semchunk budgets, and behavior when anchors are too sparse/dense are all left to a spike. This is not just a tunable number; it determines chunk boundaries and therefore reuse keys. — Bind the algorithm now: anchor predicate, max/min segment sizing, budget overflow behavior, and deterministic fallback; leave only numeric density targets for measurement.

16. [SEVERITY: minor] [plan/designs/evidence_lifecycle_design.md §6/§9/§11; plan/designs/e1_chunks_design.md §5] — The new docs still use Rule-2 deferral framing for load-bearing design choices: lifecycle §6 calls boundary-stabilized chunking a "documented next lever", §9 has an "escalation path", and E1 §5 says the embedding-model decision is "deliberately not made here" even though it controls whether the prefix stage exists. This leaves full-scope design mixed with implementation sequencing. — Recast true non-goals as boundaries, bind the full-scope mechanism for prefix/no-prefix operation, and move measurement-only sequencing language to `plan/plans/` or the spike list without "next lever" framing.

Verdict: PR #31 is not ready to merge.
Verdict: The lifecycle concepts are directionally coherent, but the schema does not yet carry the state the prose depends on.
Verdict: The blockers are executable-schema failures and missing occurrence/removal semantics.
Verdict: The major risks are partial-reuse accounting, retract timing, and batching/idempotency.
Verdict: Fix the persistence model first; polish wording and spikes after the invariants compile.
